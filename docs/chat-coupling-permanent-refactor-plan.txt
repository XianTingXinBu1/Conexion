Conexion 聊天耦合长期治理方案
================================

定位
----
本文档记录聊天模块的长期治理方向。

当前项目已经完成第一阶段治理：

1. ChatPage.vue 已经基本变成薄 View。
2. 发送消息主流程已经进入 SendMessageUseCase。
3. Prompt 构建已经有统一 usecase。
4. 会话、压缩、Prompt、生命周期已有 presentation controller / facade。
5. 架构边界已经由 npm run check:architecture 固化。

因此本文档不再描述旧的“ChatPage 巨型页面”状态，而是作为后续继续治理的路线图。


一、最终目标
------------

把聊天能力治理成边界清晰、可测试、可扩展的领域结构：

1. View 只渲染和绑定事件。
2. ViewModel 只做 composition root，不写复杂业务规则。
3. Controller / Facade 处理页面行为分组。
4. UseCase 承载业务流程。
5. Repository / Gateway 负责存储和 API 边界。
6. 核心业务逻辑可以在不挂载 Vue 组件的情况下单测。
7. 架构边界用目录结构 + 检查脚本固定，防止回潮。

当前核心分层：

View -> ViewModel -> Controller/Facade -> UseCase -> Repository/Gateway -> Storage/API

三、核心边界规则
----------------

已经通过 scripts/check-architecture-boundaries.js 检查：

1. ChatPage.vue 不直接 import:
   - composables
   - repositories
   - services
   - modules
   - api

2. chat presentation 不直接 import:
   - @/utils/storage
   - @/constants

3. chat application 不 import:
   - Vue
   - UI components
   - storage
   - notification
   - composables

命令：

  npm run check:architecture

完整健康检查：

  npm run health-check

五、当前数据流
--------------

发送消息：

ChatPage.vue
  -> useChatPageViewModel
    -> useChatSendFlow
      -> SendMessageUseCase.execute
        -> loadRegexRules
        -> apply user regex
        -> auto compression if needed
        -> create/save conversation
        -> buildSystemMessagesUseCase
        -> sendStreamChatRequest
        -> StreamMessageAssembler
        -> saveConversation
        -> UI adapter shows success/error

Prompt 构建：

useChatPromptController
  -> useChatPromptBuilder
    -> buildSystemMessagesUseCase
      -> buildSystemPrompt

后端代理：

src/api/*
  -> /api/*
    -> server/index.ts
      -> upstream OpenAI-compatible API


六、后续治理优先级
------------------

P1. 迁移 useChatSendFlow

当前：

  src/composables/useChatSendFlow.ts

建议迁到：

  src/features/chat/presentation/useChatSendFlow.ts

要求：

- 只改路径和 import。
- 不改发送行为。
- 同步调整测试。
- 验证：npm run test:run && npm run check:architecture


P2. 防止 useChatPageViewModel 膨胀

当前 ViewModel 是装配中心，这是合理的。

但需要遵守：

- 不写复杂业务规则。
- 不直接写存储 key。
- 不直接写 API 协议细节。
- 新业务优先落到 controller / facade / usecase。

如果继续增长，可拆：

  useChatPageDataSources.ts
  useChatPageControllers.ts
  useChatPageUiState.ts

但必须有明确职责，不做无意义拆分。


P3. 会话领域收口

当前会话能力分布在：

  src/composables/useConversationManager.ts
  src/composables/useConversations.ts
  src/services/conversationRepository.ts
  src/features/chat/presentation/useChatSessionFacade.ts
  src/components/ConversationListPage.vue

长期建议：

- 抽 conversation application usecases。
- 明确 conversation repository 唯一 owner。
- 让聊天页和会话列表页复用同一套会话能力。


P4. API 预设页面收口

当前 API preset 仍有页面厚度。

建议逐步拆：

- 预设 CRUD
- 表单状态
- 模型加载
- 连接测试
- 页面离开保护
- 通知

不要一次性大改。


P5. 横切 UI 统一

候选：

  ConfirmDialog
  Modal
  PageHeader
  EmptyState

目标：

- 明确通用 UI 唯一入口。
- 删除重复确认弹窗实现。
- 减少页面重复样式和重复交互。


七、禁止事项
------------

1. 禁止为了迁移目录而顺手改行为。
2. 禁止在 usecase 里调用 notification / router / DOM scroll。
3. 禁止在页面重新直接访问 storage / constants。
4. 禁止新增一坨未命名 deps 继续传来传去。
5. 禁止复制一套已有 repository / API client。
6. 禁止把 TODO 当成核心功能依赖。


八、验证策略
------------

小改动：

  npm run test:run
  npm run check:architecture

涉及类型 / 构建：

  npm run build

提交前：

  npm run health-check

无法跑完整验证时，至少说明：

- 跑了什么
- 没跑什么
- 剩余风险


九、一句话总结
--------------

聊天模块第一阶段瘦身已经完成。后续治理重点是保持边界不回潮，并把仍散落在全局 composables、页面和 services 中的聊天 / 会话专属逻辑按收益逐步收口。
